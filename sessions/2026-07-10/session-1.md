## 学习内容

### B.12 微服务架构与 API 设计

**学生初始理解**:

- 微服务解决单体太大、部署困难的问题
- 改一个业务模块不需要全部重新发版

**讲解方式**:

- 补充微服务四大收益（独立部署/独立扩展/技术栈独立/团队自治）和三大代价（分布式复杂性/运维复杂度/数据一致性）
- 强调"不是该不该拆，而是现在配不配拆"
- 讲解拆分原则：领域边界（DDD Bounded Context）、高内聚低耦合、避免循环依赖
- 对比服务间通信：同步 RPC vs 异步 MQ
- 讲解 API 版本管理：URL 版本 vs Header 版本，兼容性策略

**理解检验**:

- 按技术层拆 vs 按领域边界拆 → 初始对"技术层拆"理解有误（以为把业务模块写在一起），纠正为按技术职责抽成共享服务
- Agent 调用 LLM 用同步还是异步 → 正确判断同步，补充 SSE 流式场景
- 对话服务调用计费服务 → 正确判断异步 MQ，补充消息可靠性保障要求
- API 版本管理选型 → 倾向 Header 方案，补充 URL 方案适用场景和版本适配层概念
- 架构决策沙盘（5 个系统拆分 + 通信 + 版本管理）→ 方案合理，能按领域边界思考

**达成掌握程度**: L1
